vcpkg

  • C++版的python pip，用于管理和下载项目中的第三方依赖包，可以与CMake结合使用

Quick Start

  1. 在当前项目中git clone https://github.com/microsoft/vcpkg.git，并执行.\vcpkg\bootstrap-vcpkg.bat
    clone下来后并执行命令后，在vcpkg的目录下会产生一个可执行文件vcpkg
  1. 在与CMakeLists.txt同目录下，执行./vcpkg/vcpkg new --application，这会创建一个清单文件vcpkg.json（并开启vcpkg的清单模式），此时vcpkg会根据vcpkg.json中的依赖项自动去下载对应的第三方包，以下是一个示例
    {
  "name": "my-project",
  "version-string": "1.0.0",
  "dependencies": [
    {
      "name": "fmt",
      "version>=": "7.1.3"
    },
    {
      "name": "boost",
      "version>=": "1.75.0"
    }
  ]
  // 指定版本可能会导致下载失败，因此大多数时候可以直接 
  // "dependencies": [
  //     "fmt", 
  //     "boost"
  // ]
}
  1. 在vcpkg.json所在的目录下执行./vcpkg/vcpkg install
    或者在写好CMakeLists.txt后，先build一次，把第三方包提前下过来
    • 执行完后，会在当前目录下多出一个vcpkg_installed目录，同时在vcpkg目录中也会多出一个package目录和downloads目录，里面有下载过来的第三方包，其中：
      • vcpkg_installed目录：存放通过 vcpkg 成功安装的库的最终使用文件（头文件、库文件等）当你在项目中引用 vcpkg 安装的库时，编译器和链接器会在这个目录下查找相应的头文件和库文件。
        当把vcpkg集成到CMake里后，在cmake ..生成makefile后，在build目录里面就会多出vcpkg_installed目录
      • vcpkg/packages目录：存放每个库的安装过程中生成的中间文件和数据。
        通常，不应该直接引用packages目录中的文件来在你的项目中使用通过 vcpkg 安装的库。

        这个目录主要用于存放每个库安装过程中生成的中间文件和数据，而不是最终用于链接和编译的库文件。
      • vcpkg/downloads目录：缓存下载的所有文件，包括库的源代码等，用于加速后续安装过程。

        所以也可以手动把依赖包下载到vcpkg/downloads里，这样在vcpkg install的时候就不需要网络连接，可以直接进行离线安装

        因此当vspkg下载某个包或者文件超时，可以手动下载，然后放到downloads目录里，例如：

        发现超时下载不下来时，可以直接去对应的这个网址下载gettext-0.22.4.tar.gz，然后放到downloads目录里，并命名为gettext-0.22.4.tar.gz

  1. 编写当前项目的CMakeLists.txt，通过find_package引入第三方包。
    如果VsCode无法识别第三方头文件对应的库在哪里，从而导致IDE导致爆红：可以把vcpkg_installed/arm64-osx目录显示地添加到find_package的PATH属性里（假设是在macOS下）
    但在
    实际部署的时候不需要添加PATH属性，因为CMake会在构建Makefile时自动把第三库对应的位置加到构建文件里

    因此在写好代码并完成测试进行部署时，不添加PATH属性直接cmake .. && cmake --build .是没问题的，添加PATH属性只是为了更好写代码以及调试

CMAKE中的CMAKE_TOOLCHAIN_FILE变量

CMAKE_TOOLCHAIN_FILE 在 CMake 中是用来指定一个工具链文件的。当你在交叉编译项目时，这个变量特别有用，因为它允许你为特定目标平台配置编译器和其他工具。工具链文件包含了用于编译项目的一系列配置，如编译器的路径、目标系统的架构、编译选项和其他相关设置。

交叉编译是编译过程中的一个特殊情况，它涉及到在一个平台（宿主平台）上生成为另一个不同平台（目标平台）运行的代码。这是在开发嵌入式系统、操作系统内核、或为不同于开发机的硬件平台开发软件时常用的技术。交叉编译的关键在于使用交叉编译器，这是一种特殊的编译器，能够生成与它自身运行环境不同的目标环境代码。

通过使用 CMAKE_TOOLCHAIN_FILE，你可以轻松地将项目从一个环境切换到另一个环境，比如从开发环境切换到生产环境，或者在不同的操作系统或硬件平台之间迁移项目。这也使得在不同环境中重复使用相同的构建设置变得简单，提高了项目的可移植性和灵活性。

假设你正在为一个ARM架构的嵌入式设备进行交叉编译，你有一个专门为这个目标平台配置的工具链文件，名为arm-toolchain.cmake。这个文件可能包含了编译器的路径、编译选项以及目标设备的相关设置。你可以使用CMAKE_TOOLCHAIN_FILE来指定这个文件，以便CMake在配置项目时使用这些设置。

cmake -DCMAKE_TOOLCHAIN_FILE=/path/to/your/arm-toolchain.cmake ..

这条命令告诉CMake使用/path/to/your/arm-toolchain.cmake文件中指定的工具链来配置项目。这允许CMake生成适合于目标设备的构建文件，无论是Makefile还是其他构建系统文件。

在arm-toolchain.cmake文件中，你可能会设置诸如以下内容：

set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR arm)

set(CMAKE_C_COMPILER arm-linux-gnueabihf-gcc)
set(CMAKE_CXX_COMPILER arm-linux-gnueabihf-g++)

set(CMAKE_FIND_ROOT_PATH /path/to/your/toolchain/sysroot)

这些设置定义了系统名称、处理器类型、C和C++编译器的路径，以及CMake查找库时应该使用的根路径。通过这种方式，CMake能够正确地为目标平台配置项目，确保编译输出的可执行文件和库能够在目标设备上运行。
